# SignPath Foundation setup

The repository side of the signing pipeline is prepared. These external steps
must be completed by the repository owner after SignPath Foundation accepts the
project.

## 1. Apply

Apply at <https://signpath.org/apply> with these project details:

- Project: Hoya Agent
- Repository: <https://github.com/lihongyao517/Hoya_agent>
- License: MIT
- Code signing policy: `CODE_SIGNING_POLICY.md`
- Privacy policy: `PRIVACY.md`
- Release page: <https://github.com/lihongyao517/Hoya_agent/releases>

Enable multi-factor authentication for both GitHub and SignPath before the
application is reviewed.

## 2. Configure SignPath

Install the SignPath GitHub App for this repository and configure GitHub as the
trusted build system. The production signing policy must require origin
verification and manual approval.

Create two artifact configurations from the checked-in files:

- Application executables: `signpath/application.xml`
- NSIS and portable release executables: `signpath/release.xml`

Use the SignPath Foundation production certificate for the release signing
policy. Record the organization ID and the project, signing policy, and artifact
configuration slugs.

## 3. Configure GitHub

Create one Actions secret:

- `SIGNPATH_API_TOKEN`

Create these Actions repository variables:

- `SIGNPATH_ORGANIZATION_ID`
- `SIGNPATH_PROJECT_SLUG`
- `SIGNPATH_SIGNING_POLICY_SLUG`
- `SIGNPATH_APPLICATION_ARTIFACT_CONFIGURATION_SLUG`
- `SIGNPATH_RELEASE_ARTIFACT_CONFIGURATION_SLUG`

Do not paste the API token into an issue, commit, workflow file, or chat.

## 4. Release

1. Update `desktop/package.json` and `desktop/package-lock.json` to the same
   semantic version.
2. Merge and test the intended release commit.
3. Create and push a matching tag, for example `v1.2.0` for version `1.2.0`.
4. Approve both SignPath requests generated by the workflow.
5. Confirm that the GitHub Release contains the signed NSIS, portable, and
   Squirrel setup EXEs, the Squirrel `*-full.nupkg` and `RELEASES` files, the
   NSIS setup blockmap, and `latest.yml`.
6. Download the release and confirm both EXE signatures show `Valid` with
   `Get-AuthenticodeSignature`.
7. Confirm the workflow verifies the official Electron feed at
   `https://update.electronjs.org/lihongyao517/Hoya_agent/win32-x64/<version>/RELEASES`.

The workflow refuses ordinary branch runs and version/tag mismatches. It also
rebuilds updater metadata after the outer installer signature is applied, so
Electron Updater validates the signed bytes rather than the unsigned package.

After the first certificate is assigned, add its exact subject name to Electron
Builder's Windows `publisherName` setting and keep that value stable across
updates.
